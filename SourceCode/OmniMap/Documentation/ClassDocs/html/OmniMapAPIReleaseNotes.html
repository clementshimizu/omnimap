Release Notes for Release 2.4.0


Changes for Release 2.4.0
      
  1. OmniMap version numbering system is changed. "2.4" is the version ".0" is the minor version.
    2.   
  2. The water mark can be removed by registering Omnimap.  Please contact support@elumenati.com for more info.
    3.   
  3. New update of OmniMap should be dll compatible with previous Omnimap Releases
    4.   
  4. FEATURE REMOVED/BUG FIX: The windows registry is no longer used to set dome screen type.  Edit omnimap_user_edit.lua to change screens.
    5.   
  5. CHANGE: 
    Newly compiled Omnimap will now default to look in the follow directories:

startup directory/OmniMapConfig/
startup directory/OmniMapConfig/Resources

startup directory/OmniMapD3DConfig/
startup directory/OmniMapD3DConfig/Resources

startup directory/OmniMapD3D10Config/
startup directory/OmniMapD3D10Config/Resources
    the resource directory used to be hard coded to look in  "c:\program files\elumenati\omnimap\..." so this needs a re-compile to benifit from this change.  If it is not recompiled, it may look in the elumenati/omnimap directory.  If your program crashes after upgrading OmniMap, check the log for where it is looking for its Lua files.
    Developers can change it if they want to, but otherwise omnimap will be configured on a per installation basis.  Another benifit is that you could create two directories, "SLAMMED DEMOS" and have all of your applications ready to go on a slammed unit, and "FULLDOME" and have everything configured for that...  and breaking one will not break the rest.
    6.   
  6. Experimental integration with NAVGAR's Dome View. Please contact support@elumenati.com for more info.
    7.   
  7. Poor looking triangle tessellation.  I "tweaked" the tessellation code to add more triangles around the edge of dome, and also also the bottom edge of slammed at the cost of removing some triangle density at the dome apex.  This should look much better. Feedback is welcome.
    8.   
  8. BUG:  Since windows vista and windows 7, we can't write error logs to the program files directory during runtime.  To generate a log, run the application(for example demo.exe) with administrative privileges by right clicking on it and selecting "Run as administrator".  This will be fixed in a future minor release.
    9.   

Release Notes for Release 2.0.3


Changes for Release 2.0.3
  1. New dome screen tesselation eliminates rendering artifacts on upper right and left hand sides of dome screen output.  This new dome tesselation is now the default dome screen.
  2. Support for OmniConfig, graphical configuration tool added.
  3. Added Elumenati logo to the off screen background of the output.
Known Problems in  2.0.0:
1) Does not maintain source code compatibility with 1.x releases or previous 2.0.0 betas.  Requires a recomplilation.
2) There is not an OmniMapAppTextureD3D10 class yet.
3) We can only guarantee that OmniMap works with NVIDIA grahics hardware that has at least 128 MB of video memory.  ATI hardware has not been tested.
4) OmniMapD3D10 only works on Windows Vista with graphics cards that support DirectX 10, because DirectX 10 only works on such cards on systems with Windows Vista.

Changes for Release 2.0.0
  1. Added support for Direct3D 9 and Direct3D 10.
  2. Support for loading external shader through a Lua parameter.
  3. Reorganization of Lua scripts for increased ease of use.
  4. OmniMapAppTextureD3D and OmniMapChannelATD3D classes added for applications that want to supply their own channel rendering  in D3D.
  5. Added support for panormic shaped screen class: OM_ToroidScreen.  To learn more about screen shapes, see the OmniMap Configuration documentation.
  6. Added support for  cylinder shaped screen class: OM_CylinderScreen. To learn more about screen shapes, see the OmniMap Configuration documentation.
  7. Alpha mask support added.
  8. Run-time extensible screen geometry support added.
  9. Numerous new convenience methods added to OmniMapBase for setting and getting channel camera orientation and frustum values.  Includes removing access to OmniMapChannelBase::cam_info. All of the information previously contained in this class can be accessed via the new convenience methods.  See the OmniMapChannelBase class for more information.
  10. OmniMapAppTextureOGL, and OmniMapChannelATOGL classes added for applications that want to supply their own channel rendering.
  11. Added support for non-square non-power-of-2 channel resolutions.  Application programmes can now set an x and y dimenstion for the channel rendering canvas.  This is done in the file OmniMapConfig/omnimap_dome_wiz_ai.lua.  The line :
    12.     channelRes = 1024
    has been replaced with :
        channelResX = 1024
        channelResY = 1024
  12. To create a non-square window just set these values to the values you want to use.  Non square windows are only supported in back buffer and pbuffer rendering modes.(See #11)
  13. References to OpenGL have been removed from all base  classes.
  14. A new method for setting up the channel callbacks has been added:
      void OmniMapBase::ForEachChannel(void fun(OmniMapChannelBase *chan, void *userData), void *userData);
    This method allows the application developer to pass a void * to the channel rendering method.  This can be a reference to an object or data that is needed by the application for rendering.
  15. Numerous bug fixes.
  16. Added a watermark that is the Elumenati logo.  Please contact Elumenati, LLC at 612-605-0826 or support@elumenati.com to have the watermark removed.
